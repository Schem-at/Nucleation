# WoL Task 11 — query → pack → world (builds → world)

The inverse of `wol_extract`: given schematics (a tag-query result) + placements,
assemble them into a Minecraft world via Nucleation's streaming `WorldSink`.

## Deliverables

- **Core primitive** — `src/formats/world_pack.rs` (gated `#[cfg(not(target_arch = "wasm32"))]`,
  re-exported as `nucleation::world_pack`). Domain-agnostic: no query/tag/ORE/"wol"
  references. Two separable steps:
  - `grid_layout(items, spacing_chunks, base_y) -> Vec<Placement>` — layout.
  - `pack(placements, load, sink) -> PackStats` — the streaming packer.
- **Driver** — `examples/wol_pack.rs` (`cargo run --example wol_pack -- --out <dir> <path.schem[@x,y,z]> ...`).
- **Cargo** — `[[example]] name = "wol_pack"` (no extra features; `world_stream` is default).

## Pack + layout design

- `Placement { key: String, offset: (i32,i32,i32), local_bbox: BoundingBox }`.
  `offset` is a pure world-space translation; `local_bbox` lets the packer plan
  chunk coverage **without loading blocks**.
- **Layout** (`grid_layout`) sorts items by `key` (→ order-independent), computes a
  uniform cell stride = `max footprint (chunks) + spacing`, and chunk-aligns every
  cell origin. This **guarantees no two placements share a chunk column**, which is
  what keeps the packer's live set to a single schematic.
- **Overlap rule**: placements are totally ordered by `(key, offset)` and processed
  in that order; when two placements write the same world cell, the **later one wins**
  ("last placement wins, per block"). Deterministic and input-order-independent.

## WorldSink primitive used

`nucleation::formats::world_stream` — `WorldSink::create(dir, opts)` /
`WorldChunkView::new(cx,cz)` + `set_block` / `WorldSink::write_chunk` /
`WorldSink::finish`. Read-back for the proof uses `WorldSource::open_dir(dir).chunks()`
+ `WorldChunkView::blocks()`.

Streaming algorithm (holds **one schematic + a bounded chunk set** in RAM):
1. Plan pass (arithmetic only, no block loading): from each placement's world bbox,
   record for every chunk column the *last* processing position that touches it.
2. Process pass: for each placement in order, `load` **one** schematic, distribute
   its non-air blocks into live `WorldChunkView`s (`set_block`, later placement
   overwrites → last-wins), drop the schematic, then flush every chunk whose last
   toucher was this position (sorted → deterministic write order).

## Round-trip proof (coords asserted)

- Unit test `packs_blocks_at_placement_plus_local`: 2 schematics at distinct offsets;
  asserts each block lands at `placement.offset + local coord` after re-reading the world.
- `air_cells_are_not_written`: gaps + explicit `minecraft:air` are absent on read-back.
- `overlap_resolves_last_placement_wins`: two placements on the same cell → later key wins.
- Driver `wol_pack` re-reads the emitted world and verifies **every** placed non-air
  cell is present at its expected world coord (`ROUND-TRIP OK`). Live run over
  3 fixtures (`and`, `sample`, `test_cube`): 33 blocks written, 33 verified, 0 missing.

## Determinism + streaming evidence

- `determinism_same_inputs_same_world`: `grid_layout(items) == grid_layout(reversed items)`
  (structural layout equality) and both produce identical world block-maps.
- `streaming_live_set_is_bounded`: 9 single-chunk schematics on a grid →
  `PackStats.peak_live_chunks <= 1` (never 9). Live driver run also reported
  `peak live chunks = 1`. `peak_live_chunks` is exposed as an honest memory proxy
  (true RSS is platform-specific and intentionally not scraped).

## Emitted world format

A standard **Anvil world directory**: `region/r.X.Z.mca` + generated `level.dat`
(1.17+ layout, `status = minecraft:full` so Minecraft won't regenerate over it).
This is a real, loadable world — not a schematic-of-the-result fallback.

## Capability gap flagged

`WorldSink`/`WorldChunkView` express only a **pure block translation** — there is no
block-state *rotation* primitive at the chunk-view level. Placements therefore carry
an offset (translation) but **not** rotation. Rotating a build must be done upstream
(rotate the `UniversalSchematic` before layout). Implemented offset-only per scope
discipline; flagged here.

## Flow orchestration sketch (DOCUMENTED, not built)

Using the T10 contract (`POST /api/execute`, base64 schematic asset inputs):

1. A flow **tag-query node** queries schemati by tag → a list of schematic asset ids.
2. Flow collects each asset's bytes (base64 `.schem`) into the execute payload.
3. A new flow **`pack-world` node** calls this primitive: base64-decode each asset →
   `from_schematic` → build `(key, bbox)` items → `grid_layout` (or pass explicit
   placements from the request) → `WorldSink::create` → `pack` → `finish`.
4. The node returns/serves the world directory (tar/zip the `--out` dir) as the
   execute result; because a WoL world is a disposable cache, it can be regenerated
   on demand from the same query.

**Remaining flow-side work** (out of scope here, flow server down):
- Add the `pack-world` node type + its JSON schema (inputs: schematic assets + layout
  params; output: world archive) to the flow node registry.
- Wire the tag-query → collect-assets → pack-world edge in the flow graph runtime.
- Decide world archive transport (stream tar of `region/` + `level.dat`) and a cache
  key (hash of the sorted query result) so identical queries reuse a world.
- A thin Rust/CLI shim behind `/api/execute` that base64-decodes assets and calls
  `world_pack::pack` (the core is already binding-ready; no FFI needed for the Rust
  execute path).
