# Exact-fingerprint peak-memory reduction (output-preserving)

## Problem
`FingerprintSpec::exact()` OOM-killed (~5 GB) on one ~10M+ block dense-fill
during a 3,409-build ingest. Fix must be memory-only: the 128-bit fingerprint
value is a stable identity (3,405 live schemati fingerprints + flow/schemati
dedup), so the output must stay byte-identical.

## Root cause (profiled)
The one `fingerprint()` code path materialized, per occupied cell, **three
O(cells) buffers simultaneously** at peak, plus a per-cell token string:
- `cell_list: Vec<((i32,i32,i32), usize)>` — 24 B/cell.
- per-orbit `cells: Vec<((i32,i32,i32), Token)>` — ~36 B/cell **plus an owned
  `SmolStr` token clone per cell** (heap-allocated once the token exceeds 22
  bytes, e.g. blockstates with several properties).
- `serialize_cells() -> Vec<u8>` — the *entire* canonical byte stream
  (`~16 + token_len` bytes/cell) fully materialized before hashing.

On a dense fill (one cell per block, long redstone/log tokens) these three add
up to gigabytes; on the pathological build that reached ~5 GB and was
OS-OOM-killed. The `exact` preset uses `Symmetry::None`, i.e. a **single** orbit
element — so the cross-orbit minimization the buffers exist for is a no-op, yet
it still paid the full materialization cost.

## Change
Added `fingerprint_single_orbit()` in `src/fingerprint/mod.rs`, taken whenever
`spec.symmetry.elements().len() == 1` (the `exact` preset and any
`Symmetry::None` spec). It:
- keeps a **compact `(x,y,z, palette_id: u32, nbt_id: u32)`** record per cell —
  no per-cell token string, no `cell_list`;
- **streams** the canonical bytes (identical layout to `serialize_cells`)
  through a bounded 64 KiB buffer into `blake3::Hasher`, instead of building the
  whole `Vec<u8>`;
- reconstructs the full token (base + NBT) only on position ties in the sort, so
  ordering matches the general path exactly even with overlapping regions.

The multi-orbit fuzzy path (`shape`/`structural`/`redstone*`) is **untouched**.
No public signature, FFI/bridge, or codegen change.

## Output preservation (byte-identical)
- Golden test `exact_memory_goldens::exact_fingerprint_is_byte_identical_to_goldens`
  (`src/fingerprint/mod.rs`) captures pre-refactor exact hex for 7 fixtures
  (small, mixed palette, directional props, single + multiple block-entities,
  dense fill, two-palette dense) and asserts equality after. **Green.**
- Live big-build cross-check: the 10M-block harness produced the same hash
  `b5accce93965ac4cab524654c73527c3` on old and new code.
- `cargo test --lib fingerprint`: **31 passed, 0 failed** (incl. determinism /
  translation-invariance / block-entity / rotation-tolerant suites).

## Memory proof (`examples/fingerprint_mem.rs`, `/usr/bin/time -l`, release)
Two-block palette with multi-property tokens; peak = "maximum resident set size".

| build            | old peak RSS | new peak RSS | hash |
|------------------|-------------:|-------------:|------|
| 10,077,696 blocks | 1382.6 MB (1,449,771,008 B) | 613.3 MB (643,121,152 B) | identical `b5accce9…` |
| 27,000,000 blocks | (extrapolates >3.5 GB) | 1216.3 MB (1,275,428,864 B) | `57747f9a…` |

**~56% lower peak** on the 10M build (new is 44% of old); the delta (~770 MB) is
pure fingerprint transient — the shared schematic storage is identical in both.
The transient now scales as ~20 B/cell + a fixed 64 KiB buffer instead of
~(60–90 B/cell across three buffers), so the pathological ingest build stays
well under a sane ceiling (a 2.7×-larger 27M-block fill peaks at ~1.2 GB).

## Inherent limit
The remaining peak is dominated by the schematic's own storage plus the single
compact `cells` vector (20 B/cell), which is the minimum needed to sort into
canonical order. Sorting requires holding all cells, so it can't drop below
O(cells) without changing the output ordering — out of scope for a byte-
identical optimization. This is already an ~2.3× headroom improvement and
removes the OOM.
